# Happy Vision Pre-Dogfood Hardening Checklist

**目標：** 同事首次實測前，系統性覆蓋常見 edge case，把潛在 bug 在上線前挖出來修掉。

**規則：**
- `[ ]` 未處理 ｜ `[x]` 已覆蓋（加上 test 檔案路徑）｜ `[SKIP-需人工]` 無法自動驗證
- 一輪只處理一個項目
- 新測試放在 `tests/hardening/`
- Gemini API 一律 mock，不打真實 request
- 每個修復 commit 一次，訊息含 edge case 描述

---

## A. 檔案系統與路徑（pipeline.py / folder_watcher.py / metadata_writer.py）

- [x] A1. 路徑含空格（`/tmp/my photos/test.jpg`）能正確分析與寫入 metadata — `tests/hardening/test_a1_path_with_spaces.py`（4 tests，已覆蓋，無 bug）
- [x] A2. 路徑含中文（`/tmp/測試照片/阿公生日.jpg`）能正確處理 — `tests/hardening/test_a2_path_with_cjk.py`（3 tests，含 resume key 驗證）
- [ ] A3. 路徑含 emoji 或特殊符號能正確處理
- [x] A4. 副檔名大小寫混合（`.JPG` / `.jpg` / `.JPEG` / `.jpeg`）都能被識別 — `tests/hardening/test_a4_extension_case_mixing.py`（4 tests + scanner-agreement 回歸）
- [ ] A5. macOS HFS+ / APFS case-insensitive 檔名衝突不會重複處理
- [ ] A6. symlink 指向的 JPG 能處理，指向不存在檔案時優雅失敗
- [ ] A7. 巢狀資料夾（多層子目錄）遞迴掃描正確
- [x] A8. 空資料夾 / 只含非 JPG 檔案的資料夾 → 清楚錯誤訊息，不 crash — `tests/hardening/test_a8_empty_or_no_jpg_folder.py`（7 tests：空/非 JPG/不存在/檔案路徑/pipeline 不呼叫 API/report 空匯出）
- [ ] A9. 超長路徑（> 255 chars）處理
- [ ] A10. 唯讀資料夾 / 檔案 → metadata 寫入失敗時不污染 result_store

## B. 照片檔案本身（gemini_vision.py / metadata_writer.py / phash.py）

- [x] B1. 損壞的 JPG（檔頭錯、截斷、0 bytes）→ skip + 記錄於 result_store，不 crash — `tests/hardening/test_b1_corrupted_jpg.py`（5 tests，**修掉真實 bug**：`UnidentifiedImageError` 原本會炸掉整個批次）
- [x] B2. 偽裝副檔名（`.jpg` 實際是 PNG / HEIC / TIFF）→ 清楚錯誤，不誤判 — `tests/hardening/test_b2_fake_extension.py`（5 tests，**修掉 mime/payload mismatch bug**：`resize_for_api` 之前在小檔時 return 原 bytes，PNG 會以 `image/jpeg` 流到 Gemini）
- [x] B3. 超大尺寸（> 50MB 或 > 8000x8000）記憶體安全 — `tests/hardening/test_b3_oversized_images.py`（4 tests，含 DecompressionBomb 模擬：靠 B1 的 catch 安全失敗）
- [x] B4. 照片無 EXIF → 不 crash，metadata 照寫 — `tests/hardening/test_b4_photo_without_exif.py`（4 tests，含 byte-level APP1 segment absence check）
- [ ] B5. 照片已有 IPTC/XMP metadata 衝突 → 依設定覆蓋或保留
- [ ] B6. 同一張照片重複執行 → phash 去重有效，不重複 call Gemini
- [ ] B7. 黑白 / 去色 / 低解析度 / 純色照片 → Gemini 分析失敗時的 fallback

## C. Gemini API 與網路（gemini_vision.py / gemini_batch.py / rate_limiter.py）

- [ ] C1. 無網路 / DNS 失敗 → 清楚錯誤訊息 + 保留部分結果 + 可 resume
- [ ] C2. Gemini API timeout（模擬 30s+ 無回應）→ 重試機制啟動，最終放棄時優雅失敗
- [x] C3. 401 invalid API key → 立即停止，不繼續燒 quota，提示使用者去設定 — `tests/hardening/test_c3_invalid_api_key.py`（5 tests，**新增 `InvalidAPIKeyError` + pipeline halt**）
- [x] C4. 429 rate limit → rate_limiter 正確退避，不 crash — `tests/hardening/test_c4_rate_limit_429.py`（6 tests：retry 成功/失敗/batch 續跑 + rate_limiter refill/close 單元測試；用 mock time.sleep 讓 tests <1s）
- [x] C5. 403 quota exceeded → 停止批次，標記剩餘為 pending 可 resume — `tests/hardening/test_c5_quota_exceeded.py`（3 tests，鎖死 halt-then-resume 合約：剩餘照片保持無 row 狀態，避免 resume 死循環）
- [ ] C6. 500 / 503 server error → 重試，最終放棄時保留進度
- [x] C7. 回傳 JSON 格式錯誤 / 截斷 → parse 失敗時該張標記錯誤，不中斷整批 — `tests/hardening/test_c7_json_parse_errors.py`（12 tests，**修掉 TypeError bug**：`parse_response("[]")` 原本會炸 TypeError；現在 None / list / 空字串 / None-text 全部回傳 None）
- [ ] C8. 回傳空內容 / 拒絕分析（safety filter）→ 正確分類為失敗原因

## D. 批次處理與狀態（pipeline.py / batch_monitor.py / event_store.py / result_store.py）

- [ ] D1. 超大批次（500+ 張）記憶體穩定，無 leak
- [ ] D2. 執行中按「取消」→ 當前 inflight 的 request 能中止，資料庫狀態一致
- [x] D3. 程式崩潰後 resume → 已完成的不重跑，未完成的從中斷點繼續 — `tests/hardening/test_d3_resume_after_crash.py`（4 tests 含 idempotent 回歸 + failed 重試）
- [ ] D4. 同一資料夾同時啟動兩次分析 → 鎖定或合併，不會產生衝突狀態
- [ ] D5. 磁碟滿 → SQLite 寫入失敗時優雅處理，使用者看得懂
- [ ] D6. SQLite 檔損壞 → 偵測 + 提示備份 / 重建，不直接 crash
- [ ] D7. 執行中關閉 pywebview 視窗 → 背景任務能安全中止或完成

## E. 設定與憑證（config.py / secret_store.py / auth.py）

- [ ] E1. 首次啟動（無任何 config）→ UI 引導設定，不 crash
- [x] E2. config.json 損壞 / 格式錯誤 → fallback 到預設值 + 提示 — `tests/hardening/test_e2_corrupt_config.py`（7 tests，**修掉啟動 crash bug**：以前 corrupt config.json 直接讓 app 開不起來。現在 fallback 到 defaults + 隔離到 `config.json.bad` 給使用者救回）
- [ ] E3. API key 不存在於 Keychain → 清楚提示去哪裡設定
- [ ] E4. API key 設定後立即可用，不需重啟
- [ ] E5. 多個 API key 切換（例如免費 / 付費帳號）正確隔離
